# CI/CD Integration

> **Proposal only.** Produces a workflow file to review; makes no changes and runs nothing.
> Adding it to CI is a human decision (see
> [human-oversight-framework](../../knowledge-notes/human-oversight-framework.md)).

## Context

Generates a CI workflow that enforces the **mechanically enforceable** design-system gates on
every PR. Be honest about the boundary: the pack's LLM-driven audits don't run in CI — what
CI can enforce is the project's real tooling (the ESLint token rules, type-check, tests). This
skill operationalises the enforcement layer that `token-compliance` recommends.

## Configuration

Reads (all optional): `stack.lint.token_rules` (what lint already enforces); `stack.test`;
`integrations.*` (only to reference secret **names** if a step needs them); `target_repo`
(to detect the CI platform — defaults to GitHub Actions when `.github/` exists).

## Auto-pull integrations

None at generation time. Any integration the workflow references (Figma, Chromatic) uses
`${{ secrets.NAME }}` — never an inline value (see
[mcp-setup-guide](../../knowledge-notes/mcp-setup-guide.md)).

## Step 0 — Detect platform and gates

Default to GitHub Actions. Map gates to real commands the repo already has:
`npm run lint` (carries `no-hardcoded-tailwind-colors` / `no-raw-tailwind-typography`),
`npx tsc --noEmit`, `npm run test`.

## Step 1 — Generate the workflow (report-only first)

Emit the file with a review banner and `continue-on-error: true` on the design-system gates so
the first rollout **reports without blocking**. Once the team trusts it, they flip the flags
to blocking — the dry-run-first analogue for CI.

## Produce the proposal

```yaml
# .github/workflows/design-system.yml
# ─ GENERATED by design-system-ops · REVIEW BEFORE MERGING ─
# Report-only first: design-system gates use continue-on-error.
# Remove continue-on-error to make them blocking once trusted.
# No secrets inline — reference ${{ secrets.NAME }} only.
name: Design System

on:
  pull_request:
    paths: ["src/**/*.{ts,tsx,css}"]

jobs:
  design-system:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20", cache: "npm" }
      - run: npm ci
      - name: Lint (token + typography rules)
        run: npm run lint
        continue-on-error: true   # report-only first; remove to block
      - name: Type-check
        run: npx tsc --noEmit
        continue-on-error: true
      - name: Tests
        run: npm run test
        continue-on-error: true
```

Then a short note: what each gate enforces, which `continue-on-error` to remove first
(usually lint), and the explicit reminder that the pack's audits stay a human/agent step.

```
### Scope
- Generated: a GitHub Actions workflow proposal (report-only).
- Not done: nothing added, nothing run — you add and enable it.
- Assumed: GitHub Actions (`.github/` present); commands from package.json scripts.
- Secrets: none inline; any integration uses ${{ secrets.NAME }}.
```

## Closing note

Roll out report-only, watch a few PRs, then make lint blocking first (it's the
highest-signal, lowest-flake gate). The design-system audits in this pack remain advisory —
run them via their skills/agents, not CI.
